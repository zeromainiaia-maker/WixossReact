## 2. 起動方法

`.claude/settings.local.json` の `permissions.allow` に `Bash(codex *)` を追加済み（2026-07-19）。
Claude から直接起動できる。

```bash
codex exec --full-auto - < "<scratchpad>/codex_taskNNN.md" > "<scratchpad>/codex_taskNNN.log" 2>&1
```

- **`run_in_background: true` で起動**する（数十分かかる）。完了通知が来る。
- `--sandbox danger-full-access` は Claude Code の classifier に**ブロックされる**ことがある。`--full-auto` を使う。
- 進捗を見るときは `git status --porcelain` と `git diff --stat`、ログ末尾は `tail -c 6000 <log>`。
- ログは巨大（1〜10MB）。**全部読まない**。`grep -n "^## " <log>` でセクション位置を出してから該当箇所を読む。

### ⚠️ 自動コミットフックとの相互作用（重要）

`.claude/settings.local.json` の `PostToolUse` フックが **Write/Edit のたびに `git add -A` + commit** する。
**scratchpad（リポジトリ外）へのファイル書き込みでも発火し、リポジトリ全体をコミットする。**

- 実際に続き210 で、検証スクリプトを書いた時点で **Codex の未検証作業が `061f423b` として自動コミットされた**。
- 実害は小さい（未 push・ローカル）が、**検証時の diff ベースラインがズレる**。
- **対策**：検証を始める前に `git log --oneline -3` でベースラインとなるコミットを確定し、
  `git show <baseline>:path` で比較する。`HEAD` を使うと自分のフック起因コミットと比較してしまい**差分ゼロに見える**。

## 3. 指示書を書く前に Claude がやること（省略禁止）

### 3-1. 簿記を信用せず実測する
**PLAN の残件数は3回連続で古かった**（続き207・209・211）。必ず数え直す。


### 3-2. ベースライン数値を取る（指示書に必ず書く）
```bash

```

### 3-3. **Codex が前回言った「ブロック理由」を機械検証する**
これが最も価値を生んでいる。**2回連続で Codex の見立てが外れていた。**

- **続き210→211**：Codex は「前段が STUB だから `lastProcessedCards` に残らない」と説明したが、
  前段ステップを機械分類すると **20件は plain action（TRASH/BANISH/REVEAL/DRAW 等）で engine は記録済み**。
  実体は `LAST_PROCESSED_*` の**語彙不足**だった → 絞って投げ直し10件採用。
- **続き211→212**：Codex は「`OPTIONAL_COST` が結果を記録しない」と説明したが、ハンドラを読むと
  **エナ色コストの CHOOSE を出すだけでアクションを一切実行していない**。真因は
  **任意アクションが汎用 STUB に潰されて消えている**こと。

**やり方**＝対象効果の JSON を読み、条件の直前ステップの型を集計するワンオフを書く（`tmp_*.mjs`）。
**「Codex の説明」ではなく「コードと JSON の実測」からスコープを決める。**

### 3-4. スコープを切る
- **1バッチ 10〜30効果**。全部投げない。
- **機構が同じものだけ束ねる**。異質なものを混ぜると検証不能になる。
- 群分けして渡す（本命／既存機構で足りる分／構造が別で見送ってよい分）。

## 5. 実績のあるガードレール（なぜ入っているかの記録つき）

**そのまま流用してよい。理由を消さないこと**（消すと次に誰かが外してしまう）。

| # | ガードレール | 入っている理由（実際の事故） |
|---|---|---|
| 1 | **1効果ずつ原文照合してから条件を書く。まとめて機械置換しない** | PLAN が警告する「無検証置換で約90枚退化」の失敗モード |
| 2 | **engine の実装を読んで「本当に評価できるか」を確認してから採用** | parser が条件を吐いても engine が解決できないと**今度は常時偽＝過小実行**。続き209 で `WXK06-031` が実際に踏みかけた |
| 3 | **golden は成立/不成立の両方向を固定** | 片方向だけだと当該バグの再発を検知できない |
| 4 | **見送り判断そのものも golden で固定** | 続き209 で `WXK06-031` の非採用をテスト化＝将来うっかり採用されるのを防ぐ。良い設計なので継続させる |
| 5 | **全カード生パース diff を取り outlier 0 を機械確認**。1件でも出たら regex を絞る | regex 拡張の波及。続き208 では未申告のまま **held が40枚増**していた |
| 6 | **採用効果ごとに「逆翻訳文 _全体_ が原文と一致するか」を判定させる** | 続き209 で Codex は逆翻訳を報告に載せながら**原文との食い違いを指摘しなかった**（`WXEX1-03-E1` の `owner:'any'`、`WXK02-028-E3` の `targetsTriggerSource`）。検証側が parser の照応バグを別途発見する羽目に。**この指示を入れた続き210 で12効果、続き211 で7効果を自主申告するようになった** |
| 7 | **前回の見送り理由が今も有効か再判定させる** | 続き211 で Codex が**自分の見送り理由を2件「訂正」**（`WD20-018`/`WD21-020`）。指示に書くと機能する |
| 8 | **既存型へのフィールド追加を優先。新しい型を乱立させない** | 続き210 は新設1型のみ、続き211 は新設ゼロで達成できた |
| 9 | **JSON 手パッチは parser 同修正か MANUAL 化とセット**。`effects_*.json` はミニファイ1行＝`JSON.stringify(j)` で書き戻す | 片方だけだと held 計器が増える。pretty-print は巨大 diff |

| 13 | **`buildEffectsJson.ts` に force-adopt リストを仕込むの禁止。採用は `heldReview --adopt`（or 検証済み fresh の直パッチ）経由** | 続き251 で Codex が恒久 `FORCE_ADOPT_BOARD_ZONE_BATCH3` を build 本体に追加＝**held 計器の恒久バイパス**になり、PRESERVE（MANUAL/PARTIAL 同居）カード4枚を含む**兄弟効果12件が無審査採用**された（うち退化4件＝条件脱落2・別STUB化1・MANUAL剥がし1）。しかも本人の報告は「変化33のみ・outlier 0」＝**数値申告が外れた初のケース**（force-adopt の副作用は自分の diff 集計にも映らなかった） |

## 6. 報告フォーマット（必須項目）

**「やらなかったことの申告」を最重要と明記する。** これを書くと質が上がる。

> ⚠ **held/parserWorklist の申告値は「報告直前に `npm run build:effects` → `node scripts/heldReview.mjs` を再実行した実測値」であることを義務付ける**。続き252 で Codex は stale な held ファイルの読み値（74）を申告し、実測は 86 だった（parser 未修正の JSON 直パッチ21件ぶんのドリフトが不可視化）。

1. 触ったファイルと各1行の理由
2. **調査結果**（今回の機構が成立する前提を満たすか。効果ごとに）
3. **採用した効果の全件**＝`effectId ／ 原文の条件節 ／ 生成 JSON ／ 逆翻訳文全体 ／ 原文と一致するか`
4. **見送った効果の全件＋理由**（過去バッチの見送りの再判定結果も含める）
5. **条件以外で見つけた原文との食い違い**（0件なら「0件」と明記）
6. ゲート数値（golden / census / 同型★ / smoke / fuzz / lint warning 数 / 残件数）
7. **生パース diff の変化集合と outlier 件数**
8. **held バケットの増減**と **lint warning の増減** ← 続き208 で両方とも申告漏れがあった

## 7. Claude 側の検証チェックリスト（Codex の報告を鵜呑みにしない）

**Codex の数値申告はこれまで一貫して正確**だった。**外すのは「副作用」と「周辺の食い違い」**。そこを重点的に見る。

```bash
# 0. ベースライン確定（フック起因コミットに注意＝§2）
git log --oneline -3

# 1. ゲート独立実行
npm run gates
node scripts/groupSimilar.mjs --all        # 同型★0

# 2. JSON 変更範囲の機械 diff（申告と一致するか）
#    ※ HEAD ではなくベースラインコミットと比較する
node -e "…全 effects_*.json を <baseline> と比較して変更カードを列挙…"

# 3. held バケットの集合/枚数 diff
git show <baseline>:docs/_held_review.txt > $TEMP/h.txt
#    バケット別枚数を比較。新規増があれば中身を確認

# 4. lint の増減（ベースラインの warning 数と比較）
```

**目視で必ずやること**：
- **採用効果の条件を1件ずつ原文と突き合わせる**（閾値の向き・主語・「枚/体/種類」の区別）。
  実例：`WX06-018` は原文が3分岐（1枚/2枚/3枚→パワー3000/8000/15000）で `eq 1/2/3` が要る。
  「N**種類**」は `distinctName`、「N**枚**」は素のカウント。**ここは Codex も間違えうる**。
- **新規追加された型/フィールドを engine が実際に評価しているか**コードで確認（過小実行の検出）。
- **Codex が「一致」と書いた逆翻訳を1〜2件サンプリングして自分でも原文照合**する。

**計器で裏を取る**：held の該当バケットが狙いどおり減っているか。
例＝続き210 で `+CONDITIONAL +IS_MY_TURN` バケットが **17→4**。これが動かないなら直っていない。
