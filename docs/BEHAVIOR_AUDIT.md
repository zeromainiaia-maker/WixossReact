# 挙動トレース監査（Behavior Audit）— 実行結果 vs 原文の目視照合

> **方針**: 「JSON が原文を正しく表現しているか」を **JSON を読んで判定するのをやめ**、
> engine で**実際に効果を実行した結果（盤面差分＋ログ）**を原文と並べて**人間が目視照合**する。
> LLM を使わない。決定論・無料・回帰資産になる。
> 位置づけ＝**原文照合の主軸**。[SEMANTIC_AUDIT.md](SEMANTIC_AUDIT.md)（LLM意味比較）とは**競合ではなく補完**（behavior＝決定論・無料で盤面移動系バグに強い／semantic＝board-diffで無理なSTUB/MANUALの意味エラーに強い発見器）。**基盤は実装済み**（下記「実装状況」）。

## なぜこの方針か（背景と実測）

- **原文比較は本来かんたん**。各カードの挙動さえ見えれば人間が原文と突き合わせて判定できる。難しかったのは「JSON（DSL）を読んで挙動を想像する」工程で、とくに **STUB ノードが逆翻訳では不透明**だったこと。
- そこで「STUB が悪さをしているのでは」と疑ったが、**全10,557効果を実行して実測した結果**：
  - **素の `[STUB:]` フォールバック（本当に何もせず素通りする真の no-op）は 12 効果だけ（0.1%）**。
    固有の未配線が数枚あるのみ（`GRANT_ALL_ZONE_LIFEBURST`／`GRANT_LEAVE_PLACE_PENDING`／`LIMIT_ALL_FIELD_1`／`NEGATE_SPELL_EFFECT` 等）。
  - STUB の大半は実装済みハンドラで、ちゃんと状態を動かす（[[stub-means-implemented]] と一致）。
- **結論：STUB は犯人ではない。不透明だったのは *逆翻訳（JSON を読む）側* だけで、*実行* は普通に動く。**
  → だから「JSON を読む」のをやめ、**engine の実行結果を観る**。読むべきは JSON ではなく挙動。

## 何を作るか

`smokeTest.ts` の実行系（全効果をヘッドレスにオートパイロット実行する仕組み）を流用し、
カードごとに次を並べた**静的レビュー表（HTML）**を生成する。

| 原文テキスト | 逆翻訳（既存 `decompileEffects.ts`） | 実行前→後の盤面差分 | engine ログ（`ctx.logs`） |
|---|---|---|---|

- **ラベル付きトークンで盤面を組む**（相手シグニ甲/乙/丙・自分手札1〜5・エナ・デッキ…）。
  差分が「相手シグニ甲をバニッシュ／自分エナ+1／手札に2枚加わった／デッキを上から3枚トラッシュ」と**自然文で読める**ようにする。これが肝。
- STUB カードでも**実挙動を観測**するので、逆翻訳の不透明さが消える。
- 人間（3人バトン）は原文↔差分↔ログを見て **OK/NG** を付けるだけ。判定は機械に投げない（＝トークン0）。

## 副産物：バグの当たりが自動で付く（要レビュー・キュー）

実測で **48.3%（5,097効果）が「OK 完走したのに状態が一切変化しない」**。
多くは正当（CONTINUOUS 常時効果はモディファイア登録で盤面カード列を動かさない／条件未成立／トイ盤面に有効な対象が居ない）だが、
**先に見つかった `GRANT_PROTECTION count:'ALL'` の無言 no-op** のような**欠落バグはまさにここに潜む**。

- **非 CONTINUOUS なのに無変化かつ意味のあるログも無い効果**を要レビュー・キューに自動抽出。
- 人間は「全カード目視」ではなく「怪しい数百件から目視」に絞れる。

## 実装の中核（作り込みポイント）

1. **シナリオビルダー（最重要）**：トイ盤面だと対象不在で空振り no-op する効果が出る。
   効果の要求に応じて**有効な対象を盤に置く**（バニッシュ系→相手シグニを置く、手札操作→手札を満たす、デッキ操作→デッキを積む 等）。
   これで「空振り no-op」と「真 no-op（バグ）」を区別できる。
2. **盤面差分器**：実行前後の全ゾーン（両プレイヤー）をラベル付きで比較し、移動/増減を自然文に落とす。
3. **HTML レンダラ**：セット単位（WX01…）で原文｜逆翻訳｜差分｜ログの表を吐く。ページ送りでレビュー。
4. **要レビュー・キュー抽出**：無変化＆低情報ログの効果を CONTINUOUS 除外でリスト化。

## 進め方（段階）

1. **試作**：STUB を含む1〜2枚で挙動トレース（原文↔差分↔ログ）を出し、並びと可読性を確認。
2. **シナリオビルダー**をアクション種別ごとに拡充（バニッシュ/バウンス/トラッシュ/ドロー/エナ/デッキ操作/付与…）。
3. **HTML表**をセット単位で生成、3人バトンのレビュー資産にする。
4. **要レビュー・キュー**から欠落 no-op バグを潰す。修正は従来ルール（effects JSON 直接パッチ＋逆翻訳とエンジンのセット修正＋同型★0＋smoke/golden/fuzz、[[decompile-engine-parity]]）。
5. バグ型が見つかったら**正規表現＋JSON走査の lint**に落として全域展開（`_auditSystematicScan.mjs` の実績どおり）。回帰ゲート化。

## LLM 方式（SEMANTIC_AUDIT.md）との関係

- LLM 方式は「発見器」としては有効だったが、**固定プロンプト（ガイド約9KB）の全バッチ再送**などで完成まで高コスト。主軸から外す。
- 本方式は**トークン0・決定論・engineバグと JSON バグを同時に炙り出す**。原文照合の主軸をこちらに移す。
- LLM 方式で既に得た真バグ型（owner 取り違え76枚・GRANT_PROTECTION 48件 等）は lint 化して引き継ぐ。
