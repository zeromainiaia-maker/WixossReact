# P1完了ロードマップ — 高シグナル残作業の機械分類マップ

## このドキュメントの位置づけ
- **P1完了判定のための残作業マップ**。`npm run census` の高シグナル欠落カード（現在 BASELINE_HIGH=1447・明細 `docs/_vocab_census.txt`）を機械分類し、「parser表現で直せる純P1」「機構待ち（§6.3送り）」「両方が混在」の3区分に仕分けた結果を記録する。
- **2026-07-13（続き107）時点のスナップショット**。数字は census 再生成（`npm run census`）のたびに更新される。再現手順は末尾「再現方法」参照。
- P1完了は「census高シグナルを0にする」ことではない。PLAN §5「完了判定」の定義＝**高シグナルが逓減限界に達したら「P1完了＋残りは機構待ち」と宣言し、機構待ちを §6/§7（P2/P3）へ送る**。本ドキュメントはその宣言に向けて「残りのうち何が本当に機構待ちか／何がまだ parser 作業か」を可視化する。

## 重要な注意（caveat）
1. **粗い網（PLAN §3 原則「件数メトリクスを信じない」）**：上位カテゴリ（小さい数296・数値不一致110・クラス指定239・条件節420）は**カード単位判定**のため偽陽性を大量に含む。同カードの別効果に語彙があれば実は正しい（＝§5c 死角(b)）。過去のサンプルでは **70/73 が偽陽性**だった前例あり。**真の残作業は 1253 よりかなり少ない可能性が高い**が、正確な数は効果単位で精査するまで不明。
2. **効率の勘所**：系統カテゴリ（条件節・色・クラス・閾値）は「1 parser規則 → 多数カード」で効率的に減らせる（1バッチ 20〜50枚も可能）。これが最短ルート。手法は続き106/107で実証済み＝`src/data/effectParser.ts` の `STATE_CONDITION_CLAUSES` 表に条件を1件追加すると「〜の場合、代わりに」置換や条件持ち上げが一括で効く。
3. **粗い網カテゴリ（小さい数・数値不一致）**は偽陽性精査が主で、census を効果単位に精密化してからが本筋（別軸）。

## 再現方法（数字の更新手順）
- **入力**：`docs/_vocab_census.txt`（`npm run census` で再生成）。
- **手順**：
  1. 各 `## カテゴリ ［原文該当 X／高シグナル N／…］` の直後の `### 高シグナル（対応語彙なし）` の次行がカードリスト。
  2. カード番号へ正規化：`.replace(/\(.*$/,'').replace(/-(E\d+[a-z]?|BURST\d*)$/,'')` して重複除外。
  3. 上記「機構キーワード集合」で各カテゴリを P1／機構に二分。
  4. カードごとに「機構カテゴリのみ→純§6.3／P1カテゴリのみ→純P1／両方→混在」で集計。
- 完全なワンショットスクリプトは `docs/_p1_classification.txt` 生成時のもの（`tmp_p1class.mjs`・使い捨て済み）を上記手順から再構成可能。
